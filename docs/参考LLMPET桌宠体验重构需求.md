# 看板娘参考 LLMPET 的桌宠体验重构需求文档

版本：v0.1
状态：已确认  
创建日期：2026-07-20  
适用项目：看板娘（macOS）  
参考项目：`/Volumes/HIKSEMI/GitHub_project/LLMPET`

> 本文档用于定义“参考 LLMPET 重构看板娘”的目标、范围、交互、状态模型、迁移方式和验收标准。  
> 所有带有 `【待确认】` 的内容都存在歧义，文末集中列出了待决策项，可直接手工修改后再进入开发。

---

## 1. 文档目标

本次重构希望吸收 LLMPET 在以下方面的设计：

1. 透明、置顶、跨桌面的桌宠窗口体验。
2. 由业务状态驱动宠物素材和动画，而不再只有“站立 GIF / 点击 GIF”。
3. 将长期状态、短暂反馈和用户交互分开管理。
4. 在宠物附近展示气泡、确认卡片、状态提示和快捷操作。
5. 完善透明区域点击穿透、宠物拖动、动态窗口尺寸和位置持久化。
6. 将 View、状态协调、窗口控制和业务服务进一步解耦，避免继续扩大 `PetViewBackend`。

本次重构继续使用 SwiftUI + AppKit 原生技术栈，不将项目迁移到 Electron，也不直接复制 LLMPET 的 HTML、CSS、JavaScript 实现。

---

## 2. 背景与当前问题

### 2.1 当前已有能力

看板娘当前已经具备：

- macOS 透明悬浮窗口。
- SwiftUI + AppKit 混合窗口管理。
- AI 流式对话。
- 内置角色和自定义角色导入。
- 站立 GIF 与点击 GIF 切换。
- 输入框、输出区域和独立对话窗口。
- 模型配置、技能注入、自动化和触发器。
- 命令执行确认与安全检查。
- `PetControlService` 结构化控制入口和审计日志。
- 透明像素 Alpha 命中检测。

### 2.2 当前主要问题

1. `PetCharacter` 仅包含 `normalGif` 和 `clickGif`，无法表达思考、回复、工作、等待确认、成功、失败和睡眠等业务状态。
2. `isThinking`、`isExecutingCommand`、`showCommandConfirm` 等业务状态没有统一映射到宠物视觉状态。
3. 状态切换逻辑散落在 `PetViewBackend` 的请求、命令、自动化和触发器流程中，后续扩展容易产生冲突。
4. 输入框和输出框即使透明度为零，仍可能占据主窗口布局空间，形成较大的透明矩形区域。
5. 当前 Alpha 命中视图实现了点击和悬停，但没有看到完整的 `mouseDragged` 窗口拖动闭环。
6. 透明像素命中返回 `nil` 不等同于整个 `NSWindow` 一定能把点击交给后方应用，需要统一的窗口级穿透策略。
7. 宠物视图、输入、输出、命令确认、自动行为等职责集中度较高，不利于分别测试和演进。
8. 当前角色导入格式无法为每个状态配置不同素材。

## 3. 重构目标

### 3.1 核心目标

- 建立统一、可测试的宠物视觉状态系统。
- 让 AI 对话、命令、触发器、自动化和用户互动能够驱动宠物状态。
- 保留现有功能和用户数据，避免重构造成行为回退。
- 提升桌宠窗口的拖动、穿透、动态布局和快捷操作体验。
- 允许角色为不同状态配置不同 GIF，同时保留旧角色兼容能力。
- 为未来接入外部 Agent 状态预留明确的数据契约，但本期是否接入 Claude Code 由待确认项决定。

### 3.2 成功标准

1. 用户能够从宠物视觉上区分空闲、思考、回复、工作、等待确认、成功、失败和睡眠。
2. 状态切换只通过统一状态协调器完成，View 不自行推断业务状态。
3. 透明区域不阻挡后方应用点击，交互控件仍可正常使用。
4. 用户可以从宠物非透明区域拖动主窗口。
5. 气泡或卡片展开时，宠物在屏幕上的视觉落点不发生明显跳动。
6. 旧的内置角色、自定义角色、自动化、触发器、模型设置和技能配置仍然可用。
7. 重构后的核心状态规则具备单元测试。

## 4. 设计原则

### 4.1 借鉴原则

借鉴 LLMPET 的：

- 状态驱动视觉系统。
- 状态快照与即时事件分离。
- 状态优先级聚合。
- 状态素材映射与回退规则。
- 气泡、确认卡片和快捷菜单围绕宠物展示。
- 动态窗口尺寸和底部锚定。
- 透明区域点击穿透。
- 主宠物窗口与完整功能窗口分离。

不直接照搬：

- Electron 技术栈。
- Claude Code Hook、Token 计费、进程扫描和会话聚焦实现。
- LLMPET 的单文件渲染脚本结构。
- 未取得独立授权的第三方 GIF 素材。

### 4.2 原生实现原则

- 界面继续使用 SwiftUI。
- 透明窗口、事件命中、拖动和窗口尺寸由 AppKit 负责。
- 状态变更集中在 `@MainActor` 的状态协调器。
- 状态协调器不直接发送网络请求、不播放音频、不执行命令。
- 业务服务只发布结构化事件，不直接操作具体 View。
- `PetControlService` 继续作为机器可调用动作的稳定控制面。

## 6. 用户故事

### US-R001｜业务状态可视化

> 作为用户，我希望宠物根据 AI 和任务状态切换动作，以便不用打开窗口也能知道当前进度。

### US-R002｜等待确认提醒

> 作为用户，我希望命令需要确认时宠物给出明显提示并展示确认卡片，以免任务被静默阻塞。

### US-R003｜透明区域不挡操作

> 作为用户，我希望桌宠窗口只有宠物、输入框、气泡和菜单能够接收鼠标，透明空白处可以操作后面的应用。

### US-R004｜拖动宠物

> 作为用户，我希望从宠物本体拖动窗口，并且不会误触发点击动作。

### US-R005｜多状态角色

> 作为角色制作者，我希望给角色的不同状态配置不同素材，并能为缺失状态设置自动回退。

### US-R006｜旧角色继续使用

> 作为现有用户，我希望升级后原有内置角色和自定义角色无需重新导入即可继续使用。

### US-R007｜快速操作

> 作为用户，我希望通过宠物附近的快捷菜单进入对话、设置、角色切换、自动化或退出应用。

### US-R008｜完整对话不挤占桌面

> 作为用户，我希望桌宠气泡只展示适量内容，长对话仍在独立对话窗口中查看。

## 7. 宠物状态模型

### 7.1 基础状态

首期建议定义以下基础状态：

| 状态 | 含义 | 典型来源 | 默认回退素材 |
|---|---|---|---|
| `idle` | 空闲待命 | 没有任务 | `normalGif` |
| `thinking` | 等待模型首段回复 | AI 请求已发送、尚未收到内容 | `idle` |
| `talking` | 正在输出回复 | 收到流式响应 | `thinking` → `idle` |
| `working` | 正在执行本地动作 | 命令、触发器或自动化动作执行中 | `thinking` → `idle` |
| `waitingForConfirmation` | 等待用户确认 | 命令执行确认、安全确认 | `thinking` → `idle` |
| `success` | 操作成功的短暂反馈 | AI 回复完成、命令或触发器成功 | `idle` |
| `error` | 操作失败或异常 | 网络、解析、命令、触发器失败 | `idle` |
| `sleeping` | 长时间无活动 | 空闲超过睡眠阈值 | `idle` |

以下状态是否首期加入存在歧义：【待确认 D-003】

- `needsInput`：模型需要用户补充内容，但不是危险操作确认。
- `interacting`：用户点击宠物时播放互动动画。
- `listening`：用户正在输入或输入框聚焦。
- `automation`：自动化流程运行中的专用状态。
- `triggered`：触发器命中后的专用状态。
- `playingAudio`：本地 MP3 正在播放。

### 7.2 基础状态与短暂效果分离

为避免状态相互覆盖，建议拆成两层：

```text
PetActivityState       长期业务状态
PetTransientEffect     有时限的短暂反馈
```

建议的短暂效果：

- `clicked`
- `greet`
- `success`
- `errorPulse`
- `attention`

短暂效果到期后必须重新根据最新业务快照计算展示状态，而不是固定回到 `idle`。

### 7.3 状态优先级

建议默认优先级从高到低为：

```text
waitingForConfirmation
> error
> needsInput（若采用）
> working
> talking
> thinking
> transient success / clicked
> idle
> sleeping
```

规则：

1. 需要用户处理的状态不能被普通工作状态覆盖。
2. 用户正在确认、输入或操作菜单时，后台快照不得导致当前卡片突然关闭。
3. 短暂效果到期后必须按最新状态重算。
4. 新的高优先级状态应立即终止低优先级短暂效果。
5. 同一时间多个业务来源发出状态时，由状态协调器聚合，不由 View 决定。
6. 是否首期支持多个并行任务的聚合由【待确认 D-016】决定。

### 7.4 状态生命周期

#### AI 对话

```text
提交输入
→ thinking
→ 收到第一段流式内容
→ talking
→ 完成
→ success（短暂）
→ idle
```

失败流程：

```text
thinking / talking
→ error
→ 用户看见错误或到达错误展示时限
→ idle
```

#### 命令执行

```text
模型返回命令
→ waitingForConfirmation
→ 用户取消
→ success 或 idle【待确认 D-012】

模型返回命令
→ waitingForConfirmation
→ 用户允许
→ working
→ success / error
→ idle
```

#### 触发器与自动化

```text
开始识别意图
→ thinking【待确认 D-017】
→ 命中并执行
→ working
→ success / error
→ idle
```

#### 空闲睡眠

```text
idle 持续达到阈值
→ sleeping
→ 用户点击、输入、收到自动化事件或开始请求
→ 对应的新状态
```

默认睡眠阈值建议为 6 分钟，与 LLMPET 当前行为接近，但最终值由【待确认 D-009】决定。

## 8. 角色与动画素材需求

### 8.1 新角色模型

建议将角色模型升级为可版本化结构：

```text
PetCharacter
├── id
├── schemaVersion
├── name
├── assetsByState
│   ├── idle: [PetAnimationAsset]
│   ├── thinking: [PetAnimationAsset]
│   ├── talking: [PetAnimationAsset]
│   ├── working: [PetAnimationAsset]
│   ├── waitingForConfirmation: [PetAnimationAsset]
│   ├── success: [PetAnimationAsset]
│   ├── error: [PetAnimationAsset]
│   └── sleeping: [PetAnimationAsset]
├── interactionAssets
├── autoMessages
└── displayOptions
```

`PetAnimationAsset` 至少包含：

| 字段 | 说明 |
|---|---|
| `id` | 素材稳定标识 |
| `location` | Bundle 资源名或沙盒内文件路径 |
| `type` | GIF 或静态图片，其他格式待确认 |
| `loop` | 是否循环 |
| `weight` | 同一状态多素材轮换时的权重，首期可固定为 1 |
| `preferredDuration` | 非循环素材的可选展示时长 |

### 8.2 素材回退规则

角色没有提供目标状态素材时，必须按统一规则回退，不能显示空白。

建议规则：

```text
waitingForConfirmation → thinking → idle
working                → thinking → idle
talking                → thinking → idle
success                → interaction → idle
error                  → waitingForConfirmation → idle
sleeping               → idle
其他                    → idle
```

### 8.3 多素材轮换

- 一个状态允许配置多张素材。
- 进入状态时选择一张素材。
- 长时间处于同一状态时可按间隔轮换。
- 默认轮换间隔建议为 60 秒。
- 轮换时不得重置业务状态。
- 是否由用户配置轮换间隔由【待确认 D-005】决定。

### 8.4 旧角色迁移

旧结构：

```text
normalGif
clickGif
```

默认迁移方式：

```text
normalGif → idle
clickGif  → interactionAssets
其他状态  → 按回退规则最终使用 idle
```

要求：

1. 不删除旧的角色文件。
2. 升级后第一次读取旧数据时自动转换为新结构。
3. 迁移成功后保存新版本数据前，应保留可恢复的旧数据副本或保持解码兼容。【待确认 D-011】
4. 自定义角色达到现有 3 个上限时仍应正常迁移。
5. 角色 ID 不应再只依赖角色名称，避免重名冲突。

### 8.5 素材版权要求

- 不将 LLMPET 的“月薪喵”GIF 复制到本项目。
- 新内置素材必须确认版权或由项目自行制作。
- 用户导入素材时可以增加“素材版权由用户负责”的提示。【待确认 D-015】
- 如果复制 LLMPET 的 MIT 代码片段，必须保留其 MIT 版权和许可声明。

## 9. 主宠物窗口需求

### 9.1 窗口属性

主窗口必须：

- 无标题栏。
- 背景透明。
- 默认无阴影。
- 位于普通窗口之上。
- 可以出现在所有桌面空间。
- 支持全屏应用辅助显示。
- 不因应用失去焦点而隐藏。
- 是否隐藏 Dock 图标由【待确认 D-013】决定。

### 9.2 动态窗口尺寸

主窗口尺寸应只覆盖当前可见内容的必要区域：

- 只有宠物时使用基础尺寸。
- 显示输入框时向上扩展。
- 显示回复气泡时按内容扩展，但不得超过屏幕可用高度。
- 显示确认卡片或快捷菜单时按内容扩展。
- 内容过长时内部滚动，不继续无限放大窗口。
- 调整窗口高度时保持宠物底部的屏幕坐标尽量不变。
- 调整窗口宽度时保持宠物中心的屏幕坐标尽量不变。
- 窗口必须限制在当前显示器的 `visibleFrame` 内。

建议通过 SwiftUI `PreferenceKey` 或专用测量 View 上报内容尺寸，再由 AppKit 窗口控制器调整 `NSWindow` frame。

### 9.3 窗口位置持久化

- 用户拖动结束后保存窗口位置。
- 只保存宠物休息状态的最终位置，不保存气泡展开造成的临时 frame。
- 启动时恢复到上次位置。
- 显示器被移除、分辨率变化或位置越界时，将窗口夹到主屏幕可见区域。
- 多显示器位置是保存绝对坐标还是显示器标识 + 相对坐标，由【待确认 D-008】决定。

### 9.4 点击穿透

交互区域必须包括：

- 宠物非透明像素。
- 输入框。
- 气泡或输出框。
- 确认卡片。
- 快捷菜单。
- 关闭、允许、拒绝等按钮。

其他透明区域应把鼠标事件交给后方应用。

AppKit 没有 Electron `forward: true` 的完全等价接口，建议采用以下默认方案：

1. 使用本地和全局鼠标位置监听。
2. 定期或按 `mouseMoved` 计算鼠标是否位于有效交互区域。
3. 在透明区域切换 `NSWindow.ignoresMouseEvents = true`。
4. 鼠标回到有效区域时恢复为 `false`。
5. 拖动、输入或确认过程中强制保持 `false`。

最终技术方案由【待确认 D-007】决定。

### 9.5 宠物拖动

- 仅从宠物有效区域开始的左键拖动可以移动窗口。
- 拖动阈值建议为 4pt，低于阈值视为点击。
- 拖动过程中不能触发点击动画。
- 拖动过程中关闭快捷菜单或不再展开新气泡。
- 拖动结束后保存位置。
- 输入框、气泡文本、按钮和滚动区域不能拖动窗口。
- 拖动期间鼠标离开宠物像素范围时仍要保持拖动捕获。

### 9.6 屏幕与空间行为

- 切换桌面空间后宠物保持可见。
- 在全屏辅助空间是否显示，沿用当前 `.fullScreenAuxiliary` 行为。
- 用户切换显示器时窗口不得丢失。
- 睡眠或空闲状态不改变窗口层级。

## 10. 主宠物视图需求

### 10.1 组件拆分

建议将当前 `PetView` 拆分为：

```text
PetRootView
├── PetInputView
├── PetSpeechBubbleView
├── PetStatusIndicatorView
├── PetConfirmationCardView
├── PetCharacterView
├── PetQuickMenuView
└── PetTransientEffectView
```

要求：

- 子 View 只消费状态和发送用户意图。
- 子 View 不直接调用网络、Shell 或音频服务。
- 角色素材解析由 `PetAssetResolver` 完成。
- 窗口尺寸由统一窗口控制器处理。

### 10.2 宠物角色视图

- 根据状态协调器给出的 `renderedState` 解析素材。
- 切换素材时避免空白帧和明显闪烁。
- 使用资源 ID 控制 GIF 重播时机。
- 静态素材可以叠加 SwiftUI 位移、缩放、摇摆或呼吸动画。
- GIF 自带完整动作时，默认不再叠加大幅容器动画。
- 素材加载失败时回退到角色 `idle`，仍失败则展示应用内置占位素材。

### 10.3 输入框

默认沿用当前逻辑：

- 鼠标悬停宠物时显示输入框。
- 输入框聚焦或已有输入时保持显示。
- 鼠标离开后延迟隐藏。
- 提交后进入 `thinking`。
- 输入框获得焦点时窗口必须能接收键盘输入。
- 透明窗口穿透不得导致输入中途失去焦点。

是否改为“点击宠物后再显示输入框”由【待确认 D-002】决定。

### 10.4 回复气泡

建议把当前输出框重构为宠物气泡，但保留完整对话窗口：

- 气泡显示最新一段回复或状态提示。
- 流式输出期间实时刷新。
- 限制最大宽度、高度和展示字符数。
- 超出高度时内部滚动或截断。
- 点击气泡可延长展示时间或打开完整对话窗口。【待确认 D-004】
- 用户选择文本时不得触发拖动。
- AI 完成后默认保持一段时间再隐藏。
- 错误信息和成功提示使用语义样式区分。

### 10.5 状态提示

至少提供：

- 思考中：文本 + 三点动画或角色专用素材。
- 等待确认：高对比色和持续提醒。
- 工作中：角色工作素材或轻量循环动画。
- 睡眠：角色睡眠素材或 `z Z` 叠加效果。
- 错误：短暂抖动、红色提示或错误素材。
- 成功：短暂跳跃、闪光或成功素材。

### 10.6 命令确认卡片

建议把系统 `.alert` 改为宠物附近的自定义确认卡片：【待确认 D-012】

- 展示将执行的命令。
- 支持“执行”和“取消”。
- 长命令允许滚动和复制。
- 卡片出现时状态为 `waitingForConfirmation`。
- 未确认前不得执行命令。
- 关闭、失焦或超时是否自动取消需明确。
- 高风险命令仍由现有安全检查拒绝，不能因为 UI 重构绕过。

### 10.7 快捷菜单

建议右键宠物打开围绕宠物排列的快捷菜单。

候选菜单项：

- 打开完整对话。
- 新建对话。
- 偏好设置。
- 切换角色。
- 自动化。
- 触发器。
- 静音。
- 退出应用。

实际菜单项与顺序由【待确认 D-014】决定。

### 10.8 点击与右键行为

当前项目左键点击宠物播放角色的 `clickGif`，LLMPET 左键打开会话 HUD。

默认建议：

- 左键拖动：移动窗口。
- 右键：打开快捷菜单。
- 双击：打开完整对话窗口。【待确认 D-002】

## 11. 业务联动需求

### 11.1 AI 对话

- `submitExternalInput` 开始时发布 `thinking`。
- 收到首个非空流式片段时切换为 `talking`。
- 完成时发布成功事件，然后回到空闲。
- 网络错误、模型错误和解析错误发布 `error`。
- 新请求必须让上一次请求残留的短暂状态失效。
- 如果未来支持取消请求，取消后应回到 `idle` 或显示取消提示。【待确认 D-018】

### 11.2 命令执行

- 识别到命令后进入 `waitingForConfirmation`。
- 用户确认后进入 `working`。
- 执行成功进入短暂成功状态。
- 执行失败进入错误状态。
- 用户取消后清理待确认状态。
- 所有现有白名单、黑名单、次数限制和结果回注逻辑继续保留。

### 11.3 自动化

- 自动化到期并开始提交提示词时更新宠物状态。
- 自动化完成后使用主气泡展示摘要。
- 多个自动化同时到期时延用现有串行策略，除非【待确认 D-016】启用并行聚合。
- 一次性自动化完成后仍自动停用。

### 11.4 触发器

- 意图检测期间是否显示 `thinking` 由【待确认 D-017】决定。
- 触发器实际动作执行期间显示 `working` 或动作专用状态。
- MP3 播放开始后，是回到 `idle` 还是保持 `playingAudio`，由【待确认 D-003】决定。
- 触发失败显示 `error`，并保留现有结构化校验和调度安全规则。

### 11.5 自动陪伴行为

- 随机自动行为不得覆盖 `thinking`、`talking`、`working`、`waitingForConfirmation` 或 `error`。
- 只有 `idle` 或 `sleeping` 可以触发自动陪伴行为。
- 自动行为播放互动素材并展示随机消息。
- 自动行为结束后根据最新业务状态重算，不固定回到 `idle`。

### 11.6 PetControlService

重构后应扩展控制服务快照：

- 当前基础状态。
- 当前实际渲染状态。
- 当前状态来源。
- 状态开始时间。
- 是否有待确认事项。
- 当前角色 ID 和素材解析结果摘要。

是否允许外部调用方直接设置视觉状态由【待确认 D-010】决定。默认建议只允许受信任的内部服务发布业务事件，不提供任意状态写入接口。

## 14. 非功能需求

### 14.1 性能

- 空闲状态不应高频进行整窗口像素渲染采样。
- 鼠标命中检测应按事件触发或做节流。
- GIF 切换不得持续清空整个图片缓存。
- 长时间运行不得累积未释放 Timer、Task 或 Combine 订阅。
- 状态不变时不得重复重载同一个素材。
- 窗口尺寸变化应做去抖，避免流式文本每个 token 都调整窗口。

### 14.2 稳定性

- 素材缺失不能导致崩溃。
- 状态事件乱序不能让旧请求覆盖新请求。
- 主窗口找不到时应记录错误而不是静默失败。
- 多显示器变化后窗口必须重新进入可见区域。
- 关闭气泡或菜单不能取消后台业务任务。

### 14.3 安全

- UI 重构不得绕过命令确认和危险命令拦截。
- 外部调用不得任意伪造“已确认”。
- 用户导入文件继续使用沙盒安全访问策略。
- API Key 存储问题不在本次重构范围，但不得进一步扩大暴露面。

### 14.4 可测试性

以下逻辑不得依赖真实 NSWindow 或真实网络才能测试：

- 状态优先级。
- 状态生命周期。
- 短暂效果到期后的回落。
- 旧角色迁移。
- 素材回退。
- 异步请求 ID 防乱序。

## 15. 验收标准

### AC-R001｜AI 思考和回复状态

**Given** 用户提交一条普通 AI 消息  
**When** 请求已发出但尚未收到内容  
**Then** 宠物显示 `thinking`；收到首个流式片段后切换为 `talking`。

### AC-R002｜完成后正确回落

**Given** AI 回复完成  
**When** 成功短暂效果播放结束  
**Then** 状态协调器依据最新快照回落，不得无条件覆盖为 `idle`。

### AC-R003｜命令确认优先级

**Given** 当前存在待执行命令  
**When** 后台又出现普通思考或自动行为事件  
**Then** `waitingForConfirmation` 保持可见，确认卡片不被关闭。

### AC-R004｜透明区域穿透

**Given** 鼠标位于桌宠窗口的完全透明区域  
**When** 用户点击  
**Then** 后方应用接收到点击，桌宠不响应。

### AC-R005｜交互区域可点击

**Given** 鼠标位于宠物非透明像素、输入框、气泡、菜单或确认按钮  
**When** 用户点击  
**Then** 桌宠窗口正常接收对应事件。

### AC-R006｜拖动与点击区分

**Given** 用户从宠物非透明区域按下鼠标  
**When** 移动超过拖动阈值  
**Then** 窗口跟随移动且不播放点击动画；未超过阈值时按点击处理。

### AC-R007｜动态窗口不跳动

**Given** 宠物位于屏幕某处  
**When** 回复气泡或确认卡片展开  
**Then** 窗口向上扩展，宠物底部位置无明显跳动。

### AC-R008｜屏幕边界

**Given** 宠物靠近屏幕边缘  
**When** 展开较大的卡片  
**Then** 窗口被限制在当前屏幕可见区域，内容通过内部滚动完整访问。

### AC-R009｜旧角色迁移

**Given** 用户已有仅包含 `normalGif` 和 `clickGif` 的角色  
**When** 升级后首次启动  
**Then** 角色仍能显示、点击互动并通过回退规则响应全部新状态。

### AC-R010｜素材缺失回退

**Given** 角色没有配置 `working` 素材  
**When** 状态进入 `working`  
**Then** 按配置的回退链展示有效素材，不出现空白或崩溃。

### AC-R011｜自动行为不打断任务

**Given** 当前处于 `thinking`、`talking`、`working` 或等待确认状态  
**When** 随机自动行为计时到期  
**Then** 自动行为不播放、不覆盖当前状态。

### AC-R012｜设置与现有功能无回退

**Given** 用户升级到重构版本  
**When** 使用模型配置、技能、自动化、触发器、音乐和完整对话窗口  
**Then** 功能和数据保持可用。

### AC-R013｜状态规则测试

**Given** 项目执行单元测试  
**When** 测试状态优先级、回落、迁移和素材解析  
**Then** 全部测试通过且不依赖真实 API。

## 17. 风险与应对

| 风险 | 影响 | 应对方式 |
|---|---|---|
| AppKit 窗口穿透无法直接复刻 Electron `forward` | 鼠标回到宠物时无法及时恢复点击 | 使用全局鼠标监听，或拆分交互窗口；先做原型验证 |
| 动态窗口尺寸频繁变化 | 流式回复时窗口抖动 | 内容尺寸去抖，限制刷新频率和最大尺寸 |
| 旧角色迁移失败 | 用户自定义角色丢失 | 保留旧解码路径、迁移前备份、失败时原样回退 |
| 多个异步任务乱序 | 宠物显示错误状态 | 所有事件携带 request/run ID，忽略过期回调 |
| GIF 资源过多 | 内存升高 | 按需加载、限制缓存、状态切换时释放不再使用的资源 |
| PetViewBackend 继续膨胀 | 重构后仍难维护 | 将状态、窗口、素材和迁移拆成独立类型 |
| 自定义确认卡片替代系统 Alert | 键盘焦点和安全行为出错 | 保留系统 Alert 作为回退，先完成交互测试 |
| 素材版权不清晰 | 无法安全发布 | 只使用明确授权或自制素材，不复制第三方表情包 |
